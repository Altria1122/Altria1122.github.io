Activiti8后端部分概述

发表于2025-03-17|更新于2025-03-17|一个项目引发的突击学习流程设计器
|浏览量:

一、概述

Activiti8 是一个强大的流程管理工具，可以实现流程相关的业务，包括申请、任务发放、审批等操作。通过 Spring Boot 的 Security 进行鉴权，能够实现流程的部署、激活、挂起等功能。其核心目的是自动化实现流程图中的各个节点，对节点任务进行监控以及流程的执行。在流程测试员环节，可以规定由哪些用户或用户组负责，这些用户登录系统后就能看到当前需要处理的任务。

二、权限要求

Activiti7 及以上版本对权限有严格要求，主要涉及 GROUP_、ROLE_ACTIVITI_USER 等字段。需要将所有登录用户划分到对应的组内，暂时用 post_code（岗位代码）表示。例如，HR 人力资源部门需要添加 GROUP_HR 权限，也可以根据实际的部门（如 dept_code）进行权限划分。同时，所有用户必须拥有后一个权限，否则无法应用 Activiti 相关功能。

三、若依代码修改（添加权限）

若依本身实现了角色主导的粗粒度权限控制，规定角色可以访问的内容，对每个用户赋予一个角色。左侧是若依原有的权限列表 permissions，可以查询到所有菜单的访问权限，实现细粒度的权限发放。右侧是修改代码后新添加的权限字段，基于角色和用户组的权限，Activiti 的候选组根据 group_user 发放。

六、实例化（创建）流程

任务是以一个层次表的形式存在，表示任务分发的先后顺序。

七、任务分发

每个流程的关节点一般需要进行审批等工作，bpmn 文件中会使用 activiti:candidateUsers=“${deptLeader}” 规定当前任务的候选人/候选组。实例化流程时会使用 withVariable(“deptLeader”, “ry”) 规定处理该任务的候选人。候选人登录系统后，通过 taskRuntime.tasks()，调用当前用户信息查询对应任务，进行展示发送。

八、实例键与业务键

Activiti 实例化流程时会生成 ID 用于标记每一个流程实例，但业务键由用户提供，用于绑定唯一的业务实体。实例键由 Activiti 生成，用于标记每一个流程实例。如果业务键是唯一的，那么业务键和实例键都可以唯一指定一个流程实例。也可以不设置业务键，将 getId() 的结果绑定到业务实体上，但不推荐这样做。

九、任务执行（表单发放）

点击任务的审批按钮，应当将任务中涉及到的表单发送给前端进行处理。现阶段可以返回固定表单，这样前端会比较麻烦，但可以实现更为多样化的表单审批结构。

十、任务执行（表单填写、调用 activiti 库）

审批时前端发送来的信息中，主要需要表单对应的两个值：controlId 和 controlValue。其他部分主要对应后端的一个 domain 类，但可以简化处理。将每一对 controlId：controlValue 作为参数，放到一个哈希表中，进行后续判别和任务执行。

十一、状态监听

待审核、已批准、未通过三种常见的状态需要实时传达给前端用户。由于没有对 record 的数据库做任何更改，需要添加一个监听器，去修改数据库中的具体状态。流程会给状态 state 注入值，当任务走到这里时开始执行函数。通过 delegateExecution 获取 Instance_id 和 BusinessKey，找到对应的 domain/BorrowRecord，更新 borrow record 中的 status 字段，表示审批状态，最后更新数据库。

十二、效果演示

用户 guanyu 发起流程，用户 ry 看到任务并审批，之后 admin 看到任务。流程定义为先 ry 审批，后 admin 审批。在 Borrow record 表中记录自增索引 ID、Apply_id、Instance_id、businessKey 等信息，同时在 Borrow approval 表中填写 approval 表，主要字段包括 apply_id（用 record 中的自增 ID）、实例键 instanceId、业务键 businessKey 等。
